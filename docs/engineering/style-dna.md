# 风格 DNA

这份文档不是重复 `code-style-spec.md`。

`code-style-spec.md` 讲的是硬边界和禁止写法。
这份文档讲的是 FluxBin 代码为什么“看起来顺”，以及写新代码时应该保留什么味道。

如果要用一句话概括当前仓库的风格，它更像：

`协议内核 > 结构诚实 > 语法讨巧`

也就是说：

- 先把字节结构和边界说清楚
- 再把状态推进和错误路径写清楚
- 最后才是语法层面的简洁和好看

## 1. 先写事实，再写包装

FluxBin 不喜欢先造抽象名词，再往里塞逻辑。

它更常见的顺序是：

1. 先把真实数据结构列出来
2. 先把真实状态分支列出来
3. 再决定需不需要包一层 helper 或 builder

所以更偏好：

- `payloadLength`
- `staticByteLength`
- `offsetCache`
- `needMoreData`
- `compiledFields`

这种名字不“优雅”，但非常诚实。

## 2. 名字必须直接暴露职责

当前风格不喜欢：

- 模糊词
- 情绪词
- 自造词
- “增强版”“高级版”“优化版”这类悬空命名

更偏好：

- 动作名函数
- 结构名字段
- 能看出输入输出边界的类型名

例如：

- `compileShape`
- `decodeFrameHeader`
- `registerType`
- `readFieldAtOffset`
- `ProtocolError`

这里的共同点是：

- 看名字就知道它做什么
- 看名字就知道它属于哪一层
- 看名字就知道失败时大概会返回什么

## 3. 控制流要像状态机，不像魔术

FluxBin 的代码应该读起来很直。

常见模式是：

1. 先拿到当前 `cursor` / `bufferLength` / `compiledField`
2. 立即检查非法状态
3. 非法就返回结构化错误
4. 合法才继续推进偏移或解码

它不追求把逻辑压成最少行数。
它追求的是：

- 每个 guard 都说得清
- 每个失败点都能解释
- 每一步都能单测

## 4. 可预期失败要显式建模

FluxBin 明显偏爱 `Result`。

不是因为 `throw` 不能用。
而是因为这里的大部分失败都属于：

- 输入非法
- 半包未收齐
- `typeId` 未注册
- shape 定义超出当前支持面
- 布局与长度契约不匹配

这些都不是“程序炸了”，而是“当前状态不满足契约”。

所以更适合写成：

- `ok`
- `value`
- `error`

这种固定键集的结构化结果。

这里要保持一个约束：

- 成功分支是 `{ ok: true; value: T; error: null }`
- 失败分支是 `{ ok: false; value: null; error: E }`

不能把 `ok` 固定成 `true`，否则失败态会丢失。

## 5. 边界比复用更重要

FluxBin 不是不想复用。
它只是把“边界不能脏”放在“少写几行”前面。

所以它宁愿：

- 在 `shape` 写一套明确校验
- 在 `registry` 写一套明确治理
- 在 `stream` 写一套明确推进
- 在 `reader` 写一套明确惰性读取

也不愿意搞一个大而全的跨层 `utils`。

这也是为什么不应该出现那种显眼的：

- `helpers.ts`
- `misc.ts`
- `shared-utils.ts`

因为这类文件很容易把层之间的职责搅混。

## 6. 热路径写法要有“硬度”

这里的“硬”不是难读。
而是：

- 数据结构固定
- 字段稳定
- 分支有限
- 输入输出明确

所以热路径偏好：

- `Uint8Array`
- `DataView`
- 固定字段对象
- 整数偏移

不偏好：

- 深层对象图
- 到处挂 metadata
- 靠 `Map` / `Set` 串热路径
- 同一个数组里混很多种状态类型

## 7. 注释只解释不能从代码直接看出的事

当前风格不依赖注释来解释“这行代码在做什么”。

它更适合写三类注释：

- 为什么这样分层
- 为什么这个顺序不能变
- 为什么这里故意不支持某种写法

换句话说：

- 行为解释优先靠命名
- 设计约束才值得写注释

但在 FluxBin 当前阶段，还要额外接受一种注释：

- 文件头块注释，用来快速告诉读者这个文件负责什么、在哪一层、和哪些边界相邻

这不是和“少注释”冲突，而是为了降低第一次进入协议代码时的阅读成本。

## 8. 测试不是补丁，是规格

FluxBin 的测试不是“随便测一下能跑”。
它更像是在写小型契约：

- 给一个最小输入
- 要么得到一个确定输出
- 要么得到一个确定错误

所以测试名应非常直给：

- `decodes frame header`
- `returns need-more-data when payload is incomplete`
- `rejects duplicate typeId registration`
- `reuses cached offset for trailing field`

这种测试风格的优点是：

- 读测试就能知道当前支持面
- 读测试就能知道当前故意不支持什么
- 回归时很容易判断是功能变化还是契约破坏

## 9. 示例代码也要保持显式

即使以后加 example，风格也不应该是“把一切都塞进一个演示函数里”。

更常见的顺序应该是：

- 先定义 shape
- 再定义 registry
- 再构造 frame 或 stream 输入
- 最后明确读取结果

这样 example 本身就能当协议契约样例。

## 10. 不抢跑，不偷未来

这套风格应该保持一个自制力：

不为了显得完整，提前塞进未来能力。

例如当前还没评审完成的东西，要直接说没冻结，而不是偷偷兼容：

- `variant`
- 高层 schema 协商
- 宽松兼容升级
- 隐式对象访问

这种克制很重要，因为它避免两个坏结果：

- 用临时兼容层把主契约污染掉
- 为了“先支持一点”留下长期还债点

## 11. 写新代码时，优先保持这 12 条感觉

1. 先问职责边界，再写实现。
2. 先问数据结构是否稳定，再写语法糖。
3. 函数名用动作，字段名用结构，不写情绪词。
4. 可预期失败优先 `Result`，不要先想到 `throw`。
5. 先 guard，后主逻辑。
6. 每个分支都要能回答“保护了什么”。
7. 不用大而全 helper 跨层复用。
8. 热路径少对象图，多偏移和稳定视图。
9. 注释只解释约束，不解释显然行为。
10. 测试名直接写成能力或失败条件。
11. example 也要当协议契约样例写，不要当 UI 草图写。
12. 对当前不支持的东西，明确拒绝，不偷偷兼容。

## 12. 一句话版本

如果以后要继续保持这套味道，可以记住这句：

`直接命名，显式建模，短控制流，硬边界，少魔法。`
